# SemVer (语义化版本) 格式详解

SemVer（Semantic Versioning）是一种广泛使用的软件版本控制规范。在前端和 Node.js 生态（npm / pnpm / yarn）中被视为标准的依赖版本管理基石。

## 1. 核心版本格式

标准版本号的格式为：**`主版本号.次版本号.修订号`**（`MAJOR.MINOR.PATCH`），版本号递增规则如下：

*   **主版本号 (MAJOR)**：当你做了**不兼容的 API 修改**时（Breaking Changes）。
*   **次版本号 (MINOR)**：当你做了**向下兼容的新功能新增**时（Features）。
*   **修订号 (PATCH)**：当你做了**向下兼容的问题修复**时（Bug Fixes）。

*附加规则：主版本号为 `0`（如 `0.y.z`）表示软件处于初始开发阶段，一切都可能随时改变，公共 API 不应被视为稳定。*

## 2. 先行版本号 (Pre-release)

在修订号之后加上一个连字符 `-` 和一连串以点分隔的标识符，表示先行版本。这代表该版本不稳定，可能无法满足预期的兼容性。

常见的先行版本阶段标识符：
*   **alpha**: 内部测试版（有很多新功能，处于极其不稳定的状态，Bug 可能很多）
*   **beta**: 公测版（加入新功能，消除了严重的 Bug，但仍存在缺陷）
*   **rc (Release Candidate)**: 候选发布版（功能不再增加，只修复 Bug，几乎等于正式版）

**版本比较示例**：`1.0.0-alpha` < `1.0.0-alpha.1` < `1.0.0-beta` < `1.0.0-rc.1` < `1.0.0`

## 3. 编译元数据 (Build Metadata)

在版本号或先行版本号之后加上一个加号 `+` 和一连串以点分隔的标识符，表示编译元数据。它通常用于记录构建时间或 Commit Hash，**在判断版本高低时会被忽略**。

**示例**：`1.0.0-beta+exp.sha.5114f85`，`1.0.0+20130313144700`

---

## 4. `package.json` 中的版本范围声明

在前端工程中，依赖包通常不会锁死在一个固定版本，而是指定一个版本范围（Range）。常用的匹配符号有：

### `^` (Caret / 插入符号) - **最常用/默认**
允许**不改变最左边非零数字**的版本更新。它能在保持兼容性的前提下，尽量更新到包含最新功能和修复的版本。
*   `^1.2.3` 匹配 `>=1.2.3 <2.0.0`（允许 `MINOR` 和 `PATCH` 更新）
*   `^0.2.3` 匹配 `>=0.2.3 <0.3.0`（因为 `0` 是最左侧非零，说明还没有正式版，此时只允许 `PATCH` 更新以防破坏 API）
*   `^0.0.3` 匹配 `0.0.3`（只能精准匹配该版本）

### `~` (Tilde / 波浪号)
允许**修订号 (PATCH)** 的更新。如果指定了次版本号，则只允许修订号更新；如果没有指定，则允许次版本号更新。
*   `~1.2.3` 匹配 `>=1.2.3 <1.3.0`（只修复 bug）
*   `~1.2` 匹配 `>=1.2.0 <1.3.0`
*   `~1` 匹配 `>=1.0.0 <2.0.0`

### 其他常用标识符
*   **精准匹配**：`1.2.3` （严格锁定只能使用该版本）
*   **`*` 或 `x`**：表示任意版本。例如 `1.2.x` 相当于 `~1.2.0`。
*   **`> / >= / < / <=`**：比较运算符。例如 `>=1.2.3`。
*   **`-` (连字符范围)**：`1.2.3 - 2.3.4`，等同于 `>=1.2.3 <=2.3.4`。
*   **`||` (逻辑或)**：`^1.0.0 || ^2.0.0`，表示可以兼容 1.x 或 2.x 版本，常用于 `peerDependencies` 中。
*   **`latest`**：包在 NPM 上的最新稳定版本标签。

## 总结
SemVer 的核心意义在于建立代码生产者（第三方包作者）与代码消费者（开发者）之间的**信任契约**。通过观察版本号的变更，开发者就能立刻知道升级这个库是否安全、是否包含破坏性变更。
